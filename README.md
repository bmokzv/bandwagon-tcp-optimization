## 搬瓦工 TCP 延迟高：先搞清楚是哪种延迟

**TCP 延迟高**，简单说就是你的本机到 VPS 之间的往返时间（RTT）偏高，或者传输过程中丢包导致 TCP 不断重传，体感上就是网页加载慢、SSH 打字有延迟、跑流量掉速。

这个问题在搬瓦工用户里很常见，背后原因分两层：

**第一层：物理路由距离 + 线路质量**。从国内到美国洛杉矶走普通 163 骨干网，延迟本来就在 150-200ms 区间，晚高峰拥堵一压，轻松 250ms+。而 CN2 GIA 线路全程走中国电信的专用通道，不经过 163，所以晚高峰基本不受影响，延迟保持在 160ms 左右。

**第二层：TCP 拥塞控制算法**。默认的 TCP Cubic 算法在丢包环境下会把发送速率压得很低，哪怕网络本身可用带宽够用，传输效率也上不去。这时候开启 BBR，就能让 VPS 更聪明地探测可用带宽，减少无谓的重传等待。

所以搬瓦工 TCP 延迟高解决的优先级是：**先排查线路问题，再做 BBR 加速**。线路差，BBR 也救不了多少；线路好，BBR 加速效果才能充分发挥。

## 第一步：判断你的延迟是什么级别的问题

打开终端，ping 一下你的搬瓦工 IP：


ping your-vps-ip


根据 ping 结果对号入座：

- **白天和晚上延迟差不多**（都在 150-180ms）：正常，这是中美物理距离决定的，CN2 GIA 线路的合理区间
- **白天还行，晚上飙到 250ms+ 甚至丢包**：典型的 CN2 GT 或 163 骨干网晚高峰拥堵，需要换机房或升线路
- **全天都是 200ms+，且丢包率高**：要么是你当前机房线路不适合你的运营商，要么是需要开 BBR
- **SSH 能连上但响应特别慢，感觉卡**：大概率是 BBR 没开，TCP 传输效率低

搞清楚是哪种情况，再对症处理。

## 第二步：开启 BBR 加速——最简单的操作

BBR 是谷歌在 Linux 内核里的一套 TCP 拥塞控制算法，专门解决在有一定丢包率的网络环境下充分利用带宽的问题，同时降低 buffer 占用，减少延迟抖动。

搬瓦工的 VPS 全部是 KVM 架构，完全支持 BBR。两种方法开启：

### 方法一：直接重装带 BBR 的系统镜像

登录 KiwiVM 控制面板 → Install new OS → 选择带 `-bbr` 后缀的系统（比如 `centos-7-x86_64-bbr`）→ 点 Reload。

重装完成后 BBR 自动生效，不用手动配置。注意重装会清空数据，提前备份。

### 方法二：在现有系统里手动安装

SSH 连上 VPS，执行：

bash
wget --no-check-certificate https://github.com/teddysun/across/raw/master/bbr.sh
chmod +x bbr.sh
./bbr.sh


脚本跑完会提示重启，输入 `y` 确认。重启后验证是否生效：

bash
lsmod | grep bbr


看到 `tcp_bbr` 字样就说明开成功了。

开完之后再测一次速，通常能感觉到明显改善——尤其是在网络不太理想的环境下，传输速率提升会很明显。

## 第四步：如果换了机房还是有问题

几个额外可以试的方向：

1. **检查本地网络**：排除路由器、光猫、宽带本身的问题。可以直接用手机热点接笔记本测一次，对比差异。

2. **尝试重装系统**：偶尔有系统配置问题会影响网络表现，重装到干净的系统（选带 BBR 的版本）能排除这类原因。

## FAQ

**Q：BBR 开了，延迟没有变化，正常吗？**

A：正常。BBR 主要优化的是在有丢包率的环境下的吞吐量效率，对于纯延迟（RTT）本身的改善有限。如果你的延迟高是因为走的 163 线路晚高峰拥堵，BBR 帮不了太多，根本解法是换 CN2 GIA 线路。

**Q：搬瓦工 TCP 延迟高，是不是 VPS 本身性能问题？**

A：大概率不是。VPS CPU/内存性能影响的是计算速度，不直接影响网络 RTT。延迟高 99% 是线路问题，1% 是服务器宿主机本身网络配置。换机房比升配置更有用。

**Q：搬瓦工的 KVM 套餐机房能切换到 CN2 GIA 吗？**

A：不能。KVM 普通套餐可以在 DC2/DC3/DC4/DC8/弗里蒙特等机房之间迁移，但不包含 CN2 GIA 机房（DC6、DC9）。要用 CN2 GIA 线路必须购买对应套餐。

**Q：已经在用 CN2 GIA-E，延迟还是偏高怎么办？**
